金析智策 项目文档

1. 项目概述

金析智策（jinxizhice-platform）是一个面向金融领域的智能分析与决策支持平台。系统通过多智能体协作、RAG 检索增强生成、知识图谱等技术，为用户提供智能化的金融数据分析、知识管理和决策辅助能力。

版本 0.4.2.dev · Python ≥ 3.11 · LangGraph 多智能体 + RAG/知识图谱 + FastAPI + Vue 3

核心能力

能力模块说明
多智能体对话基于 LangGraph 状态机的多智能体系统，支持流式对话、工具调用、人工审批中断
知识库管理支持 Milvus 向量检索与 LightRAG 图增强检索，40+ 文档格式解析
知识图谱Neo4j 图数据库驱动，支持实体/关系可视化、子图查询、向量索引
RAG 评测基准数据集管理、自动评测、指标计算（召回率、准确率等）
思维导图AI 驱动的文档结构化，自动生成层级思维导图
数据看板多维度统计分析：用户活跃度、智能体使用、工具调用、知识库状态
内容审核关键词 + LLM 双模式内容安全审核
MCP 协议支持 Model Context Protocol 外部工具扩展

5. 后端架构详解

5.1 接入层（FastAPI 网关）

入口文件：backend/server/main.py，通过 Uvicorn 运行在 :5050 端口。

请求 → CORSMiddleware → AuthMiddleware → LoginRateLimitMiddleware → AccessLogMiddleware → 路由处理
中间件职责关键逻辑
CORSMiddleware跨域资源共享允许所有来源
AuthMiddleware身份认证检查公开路径白名单，非 API 路径直接放行
LoginRateLimitMiddleware登录限流POST /api/auth/token，每 IP 60s 内最多 10 次
AccessLogMiddleware访问日志记录请求处理时间和状态码

限流实现

python
# 内存级滑动窗口限流
_login_attempts: defaultdict[str, deque[float]] = defaultdict(deque)

# IP 提取优先级：X-Forwarded-For → request.client.host → "unknown"
# 登录成功（status < 400）后自动清除该 IP 的尝试记录
# 返回 429 时携带 Retry-After 头

5.2 路由层

所有路由注册在 /api 前缀下，共 9 个路由模块：

路由模块前缀职责规模
auth_router/api/auth登录·注册·用户管理中
chat_router/api/chat对话·线程·反馈·附件大（1400+行）
knowledge_router/api/knowledge知识库 CRUD·文档·查询大
graph_router/api/graph图谱列表·子图·实体中
evaluation_router/api/evaluation基准·评测中
mindmap_router/api/mindmap思维导图小
dashboard_router/api/dashboard统计·会话·反馈大
task_router/api/tasks任务列表·取消小
system_router/api/system健康·配置·日志·OCR·Provider大

5.3 鉴权体系

JWT Bearer Token 鉴权，FastAPI 依赖注入实现三级权限控制：

依赖函数权限级别说明
get_current_user可选有 Token 则解析，无 Token 返回 None
get_required_user必须登录无有效 Token 返回 401
get_admin_user管理员要求 role 为 admin 或 superadmin

公开路径（无需认证）

  • POST /api/auth/token — 登录
  • GET /api/auth/check-first-run — 首次运行检查
  • POST /api/auth/initialize — 初始化管理员
  • POST /api/auth/register — 用户注册
  • GET /api/system/health — 健康检查
  • GET /api/system/info — 系统信息

用户角色

角色权限
superadmin最高权限，系统初始化时创建
admin管理权限，可管理用户、配置、知识库等
user普通用户，仅可使用对话功能

登录安全机制

  • IP 级限流：60 秒内最多 10 次登录尝试（中间件层）
  • 账户级锁定：连续失败 10 次后锁定，时间指数递增（2^(n-10) 秒），最长 365 天
  • 密码哈希：使用安全哈希存储
  • 软删除：用户删除为逻辑删除（is_deleted 标记）

5.4 智能体系统

基于 LangGraph 状态机实现多智能体协作，是金析智策的核心。

AgentManager（单例·自动发现）
├── ChatbotAgent      — 通用对话智能体
├── DeepAgent         — 深度推理智能体
└── FinanceAgent      — 金融分析智能体

每个 Agent 内部：
BaseAgent (抽象基类)
├── get_graph()       → CompiledStateGraph (LangGraph)
├── stream_messages() → 异步流式输出
├── invoke_messages() → 同步调用
├── get_info()        → 智能体元信息
└── load_metadata()   → 从 metadata.toml 加载配置

BaseAgent 抽象基类

python
class BaseAgent:
    name = "base_agent"
    capabilities: list[str] = []          # 能力声明
    context_schema: type[BaseContext] = BaseContext

    # 核心属性
    module_name → str    # 模块名（自动推断）
    id → str             # 类名作为唯一标识
    workdir → Path       # saves/agents/{module_name}/
    checkpointer         # AsyncSqliteSaver 或 InMemorySaver

上下文系统（BaseContext）

三级配置优先级：运行时配置 > 文件配置 > 类默认值

智能体中间件

中间件职责
DynamicToolMiddleware运行时动态选择工具，预加载 MCP 工具并按上下文过滤
context_aware_prompt()根据运行时上下文动态生成系统提示词
context_based_model()根据运行时上下文动态选择 LLM 模型
AttachmentMiddleware处理文件附件，注入到消息上下文

三种智能体

智能体模块定位特色
ChatbotAgentchatbot/通用对话文件上传、可配置工具和知识库绑定
DeepAgentdeep_agent/深度推理高级推理能力，专用提示词模板
FinanceAgentfinance_agent/金融分析金融领域专属工具和提示词

工具系统

  • 内置工具：知识库查询、知识图谱查询、网页搜索（Tavily）、计算器等
  • MCP 工具：通过 langchain-mcp-adapters 接入外部 MCP 服务
  • MySQL 工具包：安全的 SQL 执行能力（含注入防护）
  • 子智能体：calc_agent 计算器子智能体

对话流程

用户消息 → ContentGuard 审核 → Agent.stream_messages()
  → LangGraph 状态机执行
    → LLM 推理
    → 工具调用（如需要）→ 人工审批中断（如配置）
    → 生成回复
  → SSE 流式推送（JSON 编码的 chunk）
  → ConversationManager 持久化

5.5 知识库系统

采用 工厂模式 + 管理器模式，支持多种后端实现的统一管理。

KnowledgeBaseManager（统一管理器）
├── KnowledgeBaseFactory（工厂，注册/创建实例）
│   ├── MilvusKB      — Milvus 向量检索
│   └── LightRagKB    — LightRAG 图增强检索
├── global_databases_meta（全局元数据）
└── async metadata lock（线程安全锁）

文件状态生命周期

UPLOADED → PARSING → PARSED → INDEXING → INDEXED
                              ↘ ERROR_PARSING    ↘ ERROR_INDEXING

两种实现

实现后端特点
MilvusKBMilvusCollection per KB，Embedding 模型配置，chunk size/overlap 可调
LightRagKBLightRAG + Neo4j图增强检索，结合知识图谱结构化信息

文档处理管线

文件上传 → MinIO 存储 → 文档解析（OCR/文本提取）→ 分块（RAGFlow 风格）→ 向量化 → 入库

5.6 知识图谱系统

通过适配器模式统一不同图数据库后端的访问接口。

GraphAdapterFactory
├── UploadGraphAdapter    — Neo4j 直接上传型（JSONL 实体导入）
└── LightRAGGraphAdapter  — LightRAG 生成型（文档自动抽取实体/关系）
    └── BaseNeo4jAdapter  — Neo4j 底层操作（Cypher 查询构建）

5.7 对话管理

ConversationManager 负责对话的完整生命周期管理。

  • 每个对话对应一个 thread_id（UUID），绑定用户和智能体
  • 支持软删除（status: active/archived/deleted）
  • 消息类型：text / tool_call / tool_result
  • 支持多模态消息（image_content 存储 base64 图片）

5.8 任务系统

Tasker 提供异步任务队列能力，用于处理耗时操作。

场景说明
文档解析大文件 OCR 解析，可能耗时数分钟
知识库索引批量文档向量化和入库
RAG 评测基准数据集评测，涉及大量 LLM 调用
基准生成自动生成评测基准数据集

5.9 评测系统

上传/生成基准数据集（JSONL）→ 执行评测 → 计算指标 → 查看结果

基准格式：{"query": "问题", "gold_chunk_ids": ["chunk_1"], "gold_answers": ["标准答案"]}

5.10 内容审核

ContentGuard 提供双模式内容安全审核：

用户输入 → 关键词检查（始终执行）→ LLM 审核（可选）→ 通过 / 拦截

5.11 文档处理插件

处理器需要 GPU部署方式适用场景
onnx_rapid_ocr否本地轻量 OCR，CPU 即可
mineru_ocr是Docker高质量文档解析
paddlex_ocr是DockerPP-StructureV3 版面分析
deepseek_ocr否API通过 SiliconFlow 调用

5.12 模型管理

  • 对话模型：通过 select_model() 选择，支持 OpenAI/DeepSeek/DashScope/SiliconFlow/自定义 Provider
  • Embedding：默认 BAAI/bge-m3，支持本地和 API 模型
  • Reranker：默认 BAAI/bge-reranker-v2-m3，通过 enable_reranker 开关控制

5.13 配置系统

配置优先级：环境变量 > TOML 文件 > 代码默认值

8. 数据模型

SQLAlchemy ORM，存储在 SQLite（saves/database/server.db）。

ER 关系

users ──1:N──> operation_logs
conversations ──1:N──> messages ──1:N──> tool_calls
conversations ──1:1──> conversation_stats
messages ──1:N──> message_feedbacks

表结构

users（用户表）

字段类型说明
idInteger PK自增主键
usernameString UNIQUE显示名称（2-20 字符）
user_idString UNIQUE登录 ID
phone_numberString UNIQUE手机号（可选）
password_hashString密码哈希
roleStringsuperadmin / admin / user
login_failed_countInteger连续失败次数
login_locked_untilDateTime锁定截止时间
is_deletedInteger软删除标记（0/1）

conversations（会话表）

字段类型说明
thread_idString(64) UNIQUE线程 ID（UUID）
user_idString(64)用户 ID
agent_idString(64)智能体 ID
titleString(255)会话标题
statusString(20)active / archived / deleted

messages（消息表）

字段类型说明
roleString(20)user / assistant / system / tool
contentText消息内容
message_typeString(30)text / tool_call / tool_result
image_contentTextBase64 图片（多模态）

tool_calls（工具调用表）

字段类型说明
tool_nameString(100)工具名称
tool_inputJSON输入参数
tool_outputText执行结果
statusString(20)pending / success / error

message_feedbacks / conversation_stats / operation_logs

  • message_feedbacks：消息反馈（rating: like/dislike + reason）
  • conversation_stats：会话统计（message_count, total_tokens, model_used）
  • operation_logs：操作审计（operation, details, ip_address）

所有时间字段统一 UTC 存储，通过 utc_now() 生成，utc_isoformat() 格式化输出。

9. 存储层架构

存储引擎用途数据类型
SQLite关系型数据用户、会话、消息、工具调用、反馈、操作日志
Milvus向量检索知识库文档 Embedding 向量，每个 KB 一个 Collection

11. 设计模式总结

设计模式应用位置说明
单例AgentManager, DBManager, Config, ContentGuard全局唯一实例
工厂KnowledgeBaseFactory, GraphAdapterFactory, DocumentProcessorFactory根据类型创建实例
模板方法BaseAgent, KnowledgeBase, BaseDocumentProcessor定义算法骨架，子类实现
策略上下文模型选择、提示词生成、KB 实现切换运行时动态选择算法
适配器GraphAdapter（统一 Neo4j/LightRAG 接口）统一不同后端接口
装饰器DynamicToolMiddleware, AttachmentMiddleware透明增强智能体能力
仓储ConversationManager, DBManager封装数据访问逻辑
门面KnowledgeBaseManager为复杂子系统提供统一接口
责任链ContentGuard（关键词 → LLM 审核）请求沿链传递
观察者SSE 流式推送事件驱动的消息推送
