## 1. 项目概述

**金析智策**（jinxizhice-platform）是一个面向金融领域的智能分析与决策支持平台。系统通过多智能体协作、RAG 检索增强生成、知识图谱等技术，为用户提供智能化的金融数据分析、知识管理和决策辅助能力。

### 核心能力

| 能力模块 | 说明 |
|---------|------|
| **多智能体对话** | 基于 LangGraph 状态机的多智能体系统，支持流式对话、工具调用、人工审批中断 |
| **知识库管理** | 支持 Milvus 向量检索与 LightRAG 图增强检索，40+ 文档格式解析 |
| **知识图谱** | Neo4j 图数据库驱动，支持实体/关系可视化、子图查询、向量索引 |
| **RAG 评测** | 基准数据集管理、自动评测、指标计算（召回率、准确率等） |
| **思维导图** | AI 驱动的文档结构化，自动生成层级思维导图 |
| **数据看板** | 多维度统计分析：用户活跃度、智能体使用、工具调用、知识库状态 |
| **内容审核** | 关键词 + LLM 双模式内容安全审核 |
| **MCP 协议** | 支持 Model Context Protocol 外部工具扩展 |

## 5. 后端架构详解

### 5.1 接入层（FastAPI 网关）

入口文件：`backend/server/main.py`

FastAPI 应用通过 Uvicorn 运行在 `:5050` 端口，中间件按以下顺序执行（注意：Starlette 中间件注册顺序与执行顺序相反）：

```
请求 → CORSMiddleware → AuthMiddleware → LoginRateLimitMiddleware → AccessLogMiddleware → 路由处理
```

#### 中间件详解

| 中间件 | 职责 | 关键逻辑 |
|--------|------|----------|
| `CORSMiddleware` | 跨域资源共享 | 允许所有来源（`allow_origins=["*"]`） |
| `AuthMiddleware` | 身份认证 | 检查公开路径白名单，非 API 路径直接放行 |
| `LoginRateLimitMiddleware` | 登录限流 | 针对 `POST /api/auth/token`，每 IP 60 秒内最多 10 次尝试，成功登录后清除计数 |
| `AccessLogMiddleware` | 访问日志 | 记录请求处理时间和状态码 |

#### 限流实现细节

```python
# 内存级滑动窗口限流
_login_attempts: defaultdict[str, deque[float]] = defaultdict(deque)

# IP 提取优先级：X-Forwarded-For → request.client.host → "unknown"
# 登录成功（status < 400）后自动清除该 IP 的尝试记录
# 返回 429 时携带 Retry-After 头
```

#### 生命周期管理

通过 `lifespan` 上下文管理器（`server/utils/lifespan.py`）在应用启动/关闭时执行初始化和清理操作。

### 5.3 鉴权体系

系统采用 **JWT Bearer Token** 鉴权，通过 FastAPI 依赖注入实现三级权限控制：

| 依赖函数 | 权限级别 | 说明 |
|----------|----------|------|
| `get_current_user` | 可选 | 有 Token 则解析用户，无 Token 返回 None |
| `get_required_user` | 必须登录 | 无有效 Token 返回 401 |
| `get_admin_user` | 管理员 | 要求 role 为 `admin` 或 `superadmin` |

#### 公开路径（无需认证）

系统维护 `PUBLIC_PATHS` 白名单，包括：
- `POST /api/auth/token` — 登录
- `GET /api/auth/check-first-run` — 首次运行检查
- `POST /api/auth/initialize` — 初始化管理员
- `POST /api/auth/register` — 用户注册
- `GET /api/system/health` — 健康检查
- `GET /api/system/info` — 系统信息

#### 用户角色

| 角色 | 权限 |
|------|------|
| `superadmin` | 最高权限，系统初始化时创建 |
| `admin` | 管理权限，可管理用户、配置、知识库等 |
| `user` | 普通用户，仅可使用对话功能 |

#### 登录安全机制

- **IP 级限流**：60 秒内最多 10 次登录尝试（中间件层）
- **账户级锁定**：连续失败 10 次后开始锁定，锁定时间指数递增（`2^(n-10)` 秒），最长 365 天
- **密码哈希**：使用安全哈希存储
- **软删除**：用户删除为逻辑删除（`is_deleted` 标记）

### 5.4 智能体系统

智能体系统是金析智策的核心，基于 LangGraph 状态机实现多智能体协作。

#### 架构总览

```
AgentManager（单例·自动发现）
├── ChatbotAgent      — 通用对话智能体
├── DeepAgent         — 深度推理智能体
└── FinanceAgent      — 金融分析智能体

每个 Agent 内部：
BaseAgent (抽象基类)
├── get_graph()       → CompiledStateGraph (LangGraph)
├── stream_messages() → 异步流式输出
├── invoke_messages() → 同步调用
├── get_info()        → 智能体元信息
└── load_metadata()   → 从 metadata.toml 加载配置
```

#### BaseAgent 抽象基类

```python
class BaseAgent:
    name = "base_agent"
    description = "base_agent"
    capabilities: list[str] = []          # 能力声明，如 ["file_upload", "web_search"]
    context_schema: type[BaseContext] = BaseContext

    # 核心属性
    module_name → str    # 模块名（自动从类路径推断）
    id → str             # 类名作为唯一标识
    workdir → Path       # saves/agents/{module_name}/
    checkpointer         # AsyncSqliteSaver 或 InMemorySaver

    # 核心方法
    get_graph()          # 构建 LangGraph 状态机（子类实现）
    stream_messages()    # 流式对话（SSE 推送）
    get_info()           # 返回名称、描述、示例、可配置项、能力列表
    get_config()         # 从文件加载上下文配置
```

#### 上下文系统（BaseContext）

三级配置优先级：**运行时配置 > 文件配置 > 类默认值**

```python
@dataclass
class BaseContext:
    thread_id: str       # 会话线程 ID
    user_id: str         # 用户 ID
    system_prompt: str   # 系统提示词
    model: str           # 使用的 LLM 模型

    # 核心方法
    from_file()          # 从 config.yaml 加载
    save_to_file()       # 持久化到 config.yaml
    get_configurable_items()  # 提取可配置项（用于前端渲染）
```

上下文支持 `Annotated` 元数据标注，自动提取为前端可配置项。

#### 智能体中间件

| 中间件 | 职责 |
|--------|------|
| `DynamicToolMiddleware` | 运行时动态选择工具，预加载 MCP 工具并按上下文过滤 |
| `context_aware_prompt()` | 根据运行时上下文动态生成系统提示词 |
| `context_based_model()` | 根据运行时上下文动态选择 LLM 模型 |
| `AttachmentMiddleware` | 处理文件附件，注入到消息上下文 |

#### 三种智能体

| 智能体 | 模块 | 定位 | 特色 |
|--------|------|------|------|
| `ChatbotAgent` | `chatbot/` | 通用对话 | 支持文件上传、可配置工具和知识库绑定 |
| `DeepAgent` | `deep_agent/` | 深度推理 | 高级推理能力，专用提示词模板 |
| `FinanceAgent` | `finance_agent/` | 金融分析 | 金融领域专属工具和提示词，数据分析能力 |

每个智能体目录包含：
- `graph.py` — LangGraph 状态机定义（节点、边、条件路由）
- `context.py` — 继承 BaseContext 的专属上下文

- `prompts.py` — 提示词模板（可选）

### 5.5 知识库系统

知识库系统采用 **工厂模式 + 管理器模式**，支持多种后端实现的统一管理。

#### 架构

```
KnowledgeBaseManager（统一管理器）
├── KnowledgeBaseFactory（工厂，注册/创建实例）
│   ├── MilvusKB      — Milvus 向量检索
│   └── LightRagKB    — LightRAG 图增强检索
├── global_databases_meta（全局元数据）
└── async metadata lock（线程安全锁）
```

#### KnowledgeBase 抽象基类

```python
class KnowledgeBase(ABC):
    # 文件状态生命周期
    UPLOADED → PARSING → PARSED → INDEXING → INDEXED
    # 异常状态：ERROR_PARSING / ERROR_INDEXING

    # 核心方法
    create_database()    # 创建知识库
    add_file_record()    # 添加文件记录
    parse_file()         # 解析文件（提取文本/分块）
    index_file()         # 索引文件（向量化/入库）
    aquery()             # 异步查询
```

#### 两种实现

| 实现 | 后端 | 特点 |
|------|------|------|
| `MilvusKB` | Milvus 向量数据库 | Collection per KB，支持 Embedding 模型配置，chunk size/overlap 可调 |
| `LightRagKB` | LightRAG + Neo4j | 图增强检索，结合知识图谱的结构化信息提升检索质量 |

#### 元数据管理

```python
# 三层元数据结构
databases_meta: {db_id: {name, description, kb_type, embed_info, created_at}}
files_meta:     {file_id: {filename, status, path, markdown_file, processing_params}}
benchmarks_meta:{db_id: {benchmark_id: {...}}}

# 持久化策略：JSON 文件 + 原子写入（临时文件 → 重命名）+ 自动备份
```

#### 文档处理管线

```
文件上传 → MinIO 存储 → 文档解析（OCR/文本提取）→ 分块（RAGFlow 风格）→ 向量化 → 入库
```

分块系统（`chunking/ragflow_like/`）支持多种文档类型解析器：
- `general.py` — 通用文档解析
- `book.py` — 书籍解析
- `laws.py` — 法律文档解析
- `qa.py` — 问答格式解析

支持的文件格式：40+ 种，包括 PDF、Word、Excel、PPT、Markdown、HTML、图片等。

```python
    get_stats()          # 图谱统计信息
```

```python
@dataclass
```

## 8. 数据模型

### 8.1 ER 关系图

```
users ──1:N──> operation_logs
```

### 8.2 表结构详解

#### users（用户表）

| 字段 | 类型 | 说明 |
|------|------|------|

## 9. 存储层架构

系统采用多存储引擎组合，各司其职：

### 9.1 存储引擎分工

| 存储引擎 | 用途 | 数据类型 |
|----------|------|----------|
| **SQLite** | 关系型数据 | 用户、会话、消息、工具调用、反馈、操作日志 |
| **Milvus** | 向量检索 | 知识库文档的 Embedding 向量，每个知识库一个 Collection |
| **Neo4j** | 图数据 | 知识图谱的实体和关系 |
| **MinIO** | 对象存储 | 上传文件、用户头像、知识库原始文档 |
| **文件系统** | 配置与状态 | TOML 配置、JSON 元数据、智能体状态、任务数据、日志 |

### 9.2 DBManager（数据库管理器）

```python
class DBManager:  # 单例模式
    engine: AsyncEngine          # 异步 SQLite 引擎
    async_session_factory        # 异步会话工厂

    async def init_db()          # 初始化数据库（建表）
    async def get_session()      # 获取异步会话
    async def run_migrations()   # 执行迁移
```

- 使用 `aiosqlite` 驱动实现全异步操作
- 数据库文件位于 `saves/database/server.db`
- 支持自动建表和增量迁移

### 9.3 MinIO 客户端

```python
class MinIOClient:
    upload_file()       # 上传文件
    download_file()     # 下载文件
    get_presigned_url() # 生成预签名 URL
    delete_file()       # 删除文件
```

- 默认 Bucket 用于存储知识库文档和用户头像
- 与 Milvus 共享同一 MinIO 实例

### 9.4 文件系统存储

```
saves/
├── config/base.toml              # 系统配置
├── agents/{agent_name}/
│   ├── config.yaml               # 智能体运行时配置
│   ├── metadata.toml             # 智能体元信息
│   └── checkpoints.db            # LangGraph 检查点（SQLite）
├── knowledge_base_data/
│   ├── databases_meta.json       # 知识库全局元数据
│   └── {kb_type}/{db_id}/        # 各知识库的数据目录
│       ├── files_meta.json       # 文件元数据
│       └── ...                   # 解析后的文档文件
├── tasks/                        # 异步任务状态
└── logs/                         # 应用日志文件
```

元数据文件采用原子写入策略：先写临时文件，再重命名覆盖，确保断电安全。

## 11. 设计模式总结

| 设计模式 | 应用位置 | 说明 |
|----------|----------|------|
| **单例** | AgentManager, DBManager, Config, ContentGuard, DocumentProcessorFactory | 全局唯一实例，避免重复初始化 |
| **工厂** | KnowledgeBaseFactory, GraphAdapterFactory, DocumentProcessorFactory | 根据类型创建不同实现的实例 |
| **模板方法** | BaseAgent, KnowledgeBase, BaseDocumentProcessor, GraphAdapter | 定义算法骨架，子类实现具体步骤 |
| **策略** | 上下文模型选择、上下文提示词生成、KB 实现切换、OCR 引擎切换 | 运行时动态选择算法 |
| **适配器** | GraphAdapter（统一 Neo4j/LightRAG 接口） | 统一不同后端的访问接口 |
| **装饰器** | DynamicToolMiddleware, AttachmentMiddleware | 透明地增强智能体能力 |
| **仓储** | ConversationManager, DBManager | 封装数据访问逻辑 |
| **门面** | KnowledgeBaseManager | 为复杂子系统提供统一接口 |
| **责任链** | ContentGuard（关键词 → LLM 审核） | 请求沿链传递直到被处理 |
| **观察者** | SSE 流式推送 | 事件驱动的消息推送 |
